# 分期生活 · AI 应用能力方案说明


| 属性   | 说明                                      |
| ---- | --------------------------------------- |
| 文档类型 | 产品方案 / 技术设计                             |
| 产品载体 | 支付宝小程序「分期生活」                            |
| 关联工程 | AI 分期助手 Demo（Next.js + Vercel + 阶梯分期算法） |
| 核心目标 | 提升用户活跃度与商品成交转化率                         |


---

## 1. 痛点与行业分析

### 1.1 分期电商决策链路中的核心痛点

分期场景的本质并非「能否购买」，而是**现金流与心理账户是否可持续**。用户在决策链路中常见卡点如下。

#### 1.1.1 利率与成本不透明

- 多期数、多商品叠加后，用户难以快速回答：**每月还多少、总利息多少、与一次性付清差多少**。
- 活动价、免息期、贴息规则等信息碎片化，认知负担大，易产生「不敢点结算」的行为。

#### 1.1.2 预算与商品组合不匹配

- 用户常以**月供上限**或**总预算**进入商城，但传统搜索/筛选多按 SKU 维度组织，缺少「在预算内凑齐某一生活场景」的路径。
- 结果：反复加购、比价、中途放弃，转化漏斗在购物车阶段损耗明显。

#### 1.1.3 对额度与还款压力的顾虑

- 用户关心：占用多少额度、是否影响其他消费、提前还款是否划算。
- 若不能**即时、可解释**地给出还款节奏与压力感知，犹豫会直接转化为流失。

#### 1.1.4 决策与支付链路割裂

- 商详页理解的规则与结算页实付不一致时，用户会产生**信任落差**，放弃支付。
- 分期规则、优惠叠加、购物车与收银台需在同一套**可解释的规则引擎**下对齐展示。

### 1.2 行业 AI 应用现状与机会点


| 维度   | 现状                        | 机会                                  |
| ---- | ------------------------- | ----------------------------------- |
| 能力形态 | 多用于客服话术、营销文案、泛推荐，与分期决策耦合浅 | 在关键触点做「预算翻译 + 方案编排」                 |
| 用户价值 | 「聊得多」不等于「买得成」             | 将口语化需求转为**可验证的分期方案与还款结构**           |
| 壁垒   | 单点聊天易复制                   | 壁垒在**商品/风控/收银规则接入**与**试算可追溯**，而非单模型 |


**结论**：在**商详—加购—结算**链路上，用 AI 衔接自然语言与确定性试算，有机会同时拉动**活跃**与**成交转化**。

## 2. 产品解决方案（核心应用）

建议聚焦 **1～2 个强耦合交易链路的 AI 模块**，避免「大而全助手」稀释价值与研发资源。

### 2.1 模块 A：智能导购 / 方案管家

#### 2.1.1 定位

理解用户「**场景 + 预算 + 偏好**」，输出**可一键加购或打包分期**的商品组合方案，并解释「为什么这样配」。

#### 2.1.2 嵌入流程


| 触点  | 能力表现                                  |
| --- | ------------------------------------- |
| 商详页 | 「按我的月供改方案」「替换更省息的同类款」                 |
| 购物车 | 多 SKU **合并月供视图**、拆单/合单建议（与额度、免息期规则对齐） |
| 结算页 | 在合规前提下提示「调整期数后的月供/总息差」，降低临门一脚放弃率      |


#### 2.1.3 用户场景示例

- **触发时刻**：用户有明确场景与预算约束（如「考研无纸化，月供别超 400」）。
- **系统行为**：解析意图 → 调用商品池与规则 → 生成**卡片式方案**（与 Demo 中 AI 方案卡片、打包分期 CTA 一致）。
- **解决问题**：从「不知道买哪几件」变为「可接受的一单方案」，缩短决策路径。

### 2.2 模块 B：分期计算专家 / 预算匹配

#### 2.2.1 定位

将自然语言预算与**阶梯分期、多 SKU 不同期数**等规则结合，输出**结构化试算结果**（月供区间、总成本、阶段还款变化）；**语言层只做解释，不编造数字**。

#### 2.2.2 嵌入流程


| 触点  | 能力表现                               |
| --- | ---------------------------------- |
| 商详页 | 「帮我算 6 期 vs 12 期差多少」               |
| 结算页 | 多商品不同免息期合并后的**阶段还款表**（与「阶梯分期」算法对齐） |


#### 2.2.3 用户场景示例

- **触发时刻**：用户已选多件商品，期数不一致或存在「前低后高」阶梯结构。
- **系统行为**：**确定性试算服务**计算结果 → AI 解读差异与风险提示。
- **解决问题**：消除「算不清、不敢付」的心理，提升信任与结算完成率。

### 2.3 模块关系与统一入口

- **模块 A**：负责「**选什么**」。
- **模块 B**：负责「**算得准、说得清**」。
- **统一入口**：在小程序内通过悬浮球 / 固定入口分流至「聊方案」与「算分期」（与现有 Demo 中 AI 浮层入口产品形态一致）。

## 3. 设计思路与选型逻辑

### 3.1 功能取舍：为何选择「方案管家 + 计算专家」


| 评估维度     | 说明                                              |
| -------- | ----------------------------------------------- |
| **业务收益** | 直接作用于加购率、客单价、结算完成率；与「分期生活」品类心智一致。               |
| **实现成本** | 大模型主要消耗在**意图解析与话术生成**；金额与期数由规则与试算服务输出，便于控本、控延迟。 |
| **差异化**  | 相对纯推荐流，「预算 → 组合 → 可执行结算」闭环更难复制；壁垒在数据与规则接入。      |


### 3.2 AI 技术选型：大模型与传统规则引擎

#### 3.2.1 分工原则


| 层级    | 推荐技术               | 职责                    |
| ----- | ------------------ | --------------------- |
| 理解与表达 | **大语言模型（LLM）**     | 意图理解、多轮澄清、方案叙述、对比话术   |
| 数字与规则 | **规则引擎 / 确定性试算服务** | 分期试算、贴息规则、额度校验、合规话术模板 |


#### 3.2.2 选型结论

- **混合架构**：LLM 做「理解与编排」，**核心数字仅来自试算引擎**（Demo 中阶梯分期逻辑即此类能力的雏形）。
- **前端呈现**：对数字区块采用「只读渲染」，与 AI 文案区块在样式与交互上区分，强化用户心智中的**可信区**。

### 3.3 精确性、幻觉与金融合规

#### 3.3.1 计算精确性

- 所有还款额、总利息、阶段切换点**仅由后端（或边缘函数）按合同与活动规则计算**。
- LLM 输出前注入**结构化试算 JSON**；禁止模型自由生成金额字段。

#### 3.3.2 意图自然性与信息不足策略

- LLM 负责「问清约束」「解释方案权衡」；信息不足时**先追问再调接口**，避免猜测 SKU 或期数。

#### 3.3.3 幻觉风险与合规控制


| 手段     | 说明                        |
| ------ | ------------------------- |
| 模板化输出  | 关键结论采用固定模板 + 变量插槽         |
| 强制免责声明 | 如「以收银台/合同为准」              |
| 策略拦截   | 对「保证最低息」等敏感表述用词表或规则禁用     |
| 审计日志   | 留存用户问句、试算入参、结果版本号，支撑纠纷与稽核 |

## 4. 技术实现闭环（结合现有 Demo）

### 4.1 架构映射


| 产品能力        | Demo 中已有/相近能力              | 演进方向                                                   |
| ----------- | -------------------------- | ------------------------------------------------------ |
| 自然语言进线      | AI 对话面板、快捷 Prompt          | 接入支付宝小程序 **JSAPI / 开放能力**（用户标识、营销标签脱敏后传入）              |
| 方案卡片        | LLM / 场景 Fallback 生成商品组合卡片 | 卡片 CTA 与**真实商品 ID、库存、收银台**对齐；失败时降级为规则模板                |
| 阶梯 / 混合期数试算 | 结算 Sheet、分期明细              | 抽离为 **独立试算 API**（如 `installment-preview`），AI 层只消费 JSON |
| 会话与回访       | 前端 localStorage 历史         | 生产环境改为**服务端会话 + 合规留存策略**；Demo 保留用于 H5 验证与面试演示          |


### 4.2 部署与运行时（Next.js + Vercel）

- **Vercel**：适合作为 **H5 / 运营预览 / A/B 实验** 宿主，与 Git 联动迭代快。
- **支付宝小程序生产流量**：建议最终落在**合规备案域名 + 国内可用部署**，或与小程序云、自建网关对接**同一试算服务**，避免地域、合规与延迟风险。

### 4.3 前端交互与即时反馈


| 技术点   | 说明                                                  |
| ----- | --------------------------------------------------- |
| 分层反馈  | 发送后立即**乐观 UI**（骨架屏 / 进度文案）；**试算返回后再替换数字区**，避免先错后改   |
| 流式与渐进 | 方案标题、卖点可流式输出；**价格、期数、月供等字段整包展示**，减少闪烁与不信任           |
| 短闭环编辑 | 「换一个」「调期数」与现有卡片交互一致，提升活跃与转化                         |
| 弱网与小包 | 关键 CSS、同源静态资源策略（与 Demo 中关键样式、商品图策略一致）；小程序侧配合分包与 CDN |
| 数据闭环  | 从「打开 AI → 生成方案 → 加购 → 结算成功」全链路埋点，验证 AI 对活跃与转化的实际贡献  |


### 4.4 与 Demo 仓库的对应关系（便于落地对照）

- **前端框架**：Next.js（App Router），便于 SSR/静态资源与 API Route 同仓。
- **展示与交互**：商详/瀑布流/结算 Sheet 与 AI 面板同页编排，验证「触点嵌入」可行性。
- **算法资产**：阶梯分期、多 SKU 期数差异等逻辑应逐步**服务化**，供小程序与 H5 共用。

---

## 5. 文档小结

本方案将 AI 定位为**分期决策链路上的编排层**：以**确定性试算为唯一真源**，在商详、购物车、结算等触点提供可执行、可解释、可审计的能力；与当前 **Next.js + Vercel 演示前端 + 阶梯分期算法** 同构，可渐进接入支付宝小程序生产环境，并通过埋点验证对**活跃度**与**成交转化率**的拉动效果。
